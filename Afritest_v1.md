**AfriTest**

African App & Web Testing Platform

# **1\. Context & Origin**

AfriTest was conceived after observing a WhatsApp group dedicated to App Store / Play Store testing in Africa and the diaspora. The group had significant potential - a large pool of African testers across multiple countries - but lacked structure: developers dropped apps in the chat, testers responded informally, and feedback got lost in conversation noise.

The founder of AfriTest identified this as an opportunity to build a structured, professional alternative: a dedicated platform where app and web testing is organized, traceable, and valuable - for both developers and testers.

# **2\. Platform Objective**

AfriTest aims to become the reference platform for quality app and web testing in Africa, by:

- Creating a verified, structured community of African testers
- Providing developers (indie and startup) with real, actionable test reports
- Enforcing a professional QA process rather than informal group chat feedback
- Building a reputation system that values quality contributions from testers
- Flagging and scrutinizing vibe-coded apps (AI-generated code without proper oversight) for security risks
- Evolving into a talent marketplace where African recruiters find vetted tech profiles

The platform is 100% free at launch. Monetization will be considered at a later stage based on traction.

# **3\. Key Concepts & Rules**

## **3.1 Single User Model**

There are no fixed roles. Every registered user is both a potential developer and a potential tester. The system tracks each user's activity on both sides:

- Submissions: apps/sites they submitted for testing
- Tests performed: apps they tested and reported on

One hard constraint: a user cannot test their own submission.

## **3.2 Open vs. Closed Tests**

- Open test: any registered tester can self-enroll and submit a report
- Closed test: the developer restricts access to a specific list of testers (invited or admin-assigned)

For closed tests, the platform can generate and deliver a curated tester list on demand, filtered by criteria (device type, OS, country, experience level, etc.).

## **3.3 Structured QA Process**

No free-text chaos. Every test report follows a standardized form including:

- Device information (model, OS version, screen size)
- Test environment (real device vs. emulator, browser, connectivity)
- Features tested (checklist based on dev-defined scope)
- Bugs found (severity, reproduction steps, screenshot)
- Overall rating and qualitative feedback

## **3.4 Vibe Code Policy**

Applications generated by AI tools ("vibe coding") without proper security review will be flagged during submission. Flagged apps undergo additional scrutiny and are labeled accordingly in the platform. Testers are alerted to pay special attention to data handling, permissions, and security behaviors.

## **3.5 WhatsApp Bot**

A bot connected to the existing WhatsApp community notifies members when a new app is available for testing. The bot does NOT receive test feedback - all structured responses happen on the platform only.

# **5\. App Submission Process**

Any registered user can submit an app or website for testing. Publication is immediate - no prior admin approval is required. However, admins can remove or suspend a submission at any time if it violates platform rules (security risk, inappropriate content, vibe-coded app without proper disclosure, etc.).

## **5.1 Submission Form (Multi-Step)**

The submission is a guided multi-step form to ensure all necessary information is collected regardless of app type.

**Step 1 - General Info**

- App / site name
- Short description: what it does, target audience
- Type: Mobile App or Web App
- Platform: Android / iOS / Both / Web
- Is this app AI-generated (vibe coded)? Yes / No / Partially - triggers a security review label if yes

**Step 2 - Access & Links**

This step adapts based on the app type selected in Step 1:

- Web App: public URL to test
- Mobile (public): Play Store or App Store link
- Mobile (closed / internal test): APK download link OR Google Play internal test link / TestFlight link + tester email invitations required (see 5.2)
- Test credentials: optional demo account login/password if the app requires authentication

**Step 3 - Test Scope**

- List of features to test (dev defines what testers should focus on)
- Specific scenarios or user flows to cover
- Known issues or areas to ignore
- Expected number of testers (indicative)
- Desired deadline for test reports

**Step 4 - Test Type**

- Open test: any registered tester can self-enroll and submit a report
- Closed test: access is restricted; dev selects testers manually or via criteria (see 5.3)

## **5.2 Closed Tests - Google Play / TestFlight Invitations**

When a dev submits a closed mobile app (Google Play internal testing or Apple TestFlight), testers must be invited by email before they can install and access the app. AfriTest handles this as follows:

- After selecting testers (see 5.3), the platform displays the confirmed list of tester emails
- The dev copies this list and pastes it into Google Play Console or App Store Connect to send official invitations
- Testers receive the Google / Apple invitation email, accept it, and install the app
- Once access is granted, testers return to AfriTest to submit their structured report

Important: AfriTest does not send Google or Apple invitations directly. This action is performed by the dev in their own developer console. The platform generates the correct tester email list and tracks assignment status.

## **5.3 Closed Tests - Tester Selection**

The dev has two ways to select testers for a closed test, and both can be combined:

**Mode A - Manual Selection**

- Dev browses verified tester profiles on the platform
- Filters available: country, device type, OS version, language, experience level
- Dev selects profiles one by one and adds them to the test session
- Selected testers receive an in-app notification and optional email alert

**Mode B - Criteria-Based Generation**

- Dev defines criteria: e.g. 10 testers, Android, Benin or Togo, mid-range device
- The platform automatically generates a matching list from available and active testers
- Dev reviews the generated list and can swap individual profiles before confirming
- Platform exports the final tester list (name + email) for use in Google Play Console or TestFlight

## **5.4 Submission Status Flow**

| **Status**  | **Meaning**                                                    |
| ----------- | -------------------------------------------------------------- |
| Draft       | Dev started the form but has not submitted yet                 |
| Published   | Live - testers can self-enroll (open) or are notified (closed) |
| In Progress | At least one tester has submitted a report                     |
| Completed   | Deadline passed or dev manually closes the test                |
| Suspended   | Removed by admin - dev is notified with the reason             |

# **8\. WhatsApp Bot Integration**

The bot acts as a one-way broadcaster from the platform to the existing WhatsApp community. It does not process replies.

**Trigger events**

- New app submitted and approved → bot sends app name, type, platform, and platform link
- Reminder 24h before test deadline (if applicable)

**Implementation**

- Python service using WATI API or Green API
- Laravel Job/Queue: WhatsApp notification dispatched via queued job
- Message template pre-approved by WhatsApp Business
- Rate limiting to avoid spam: max 1 message per submission, batched if multiple in short window
